# 问题排查：测地线与切片错误的根因分析

**涉及模块**：`HGP_3D_Convex_Hulls_C2`、`HGP_Slicer_Mesh`、`HGP_Shortest_Geodesic_Path_C3`、`vis/backend/app.py`、`vis/backend/web/index.html`  
**日期**：2026-03-31

---

## 一、最初发现的问题现象

### 问题1：测地线自动吸附到顶点

计算两点之间的最短测地路径时，无论用户点击网格的哪个位置，起点和终点都会自动跳到最近的网格顶点，而不是精确的点击位置。

### 问题2：切片轮廓出现长连线

对随机生成的凸包进行切片时，切片轮廓线中出现了不应存在的长斜线，横跨网格两端，视觉上像是把本不相邻的轮廓段强行连接在了一起。

---

## 二、第一阶段：错误的补丁修复（治标不治本）

### 针对测地线的补丁

**判断**：CGAL 测地线算法要求点必须在网格上，任意空间点会导致崩溃，因此需要把点吸附到最近顶点。

**补丁代码（前端）**：
```javascript
// 遍历所有顶点，找最近顶点
let closestVert = null;
let minDist = Infinity;
const posAttr = currentMesh.geometry.attributes.position;
for (let i = 0; i < posAttr.count; i++) {
    const vx = posAttr.getX(i);
    const vy = posAttr.getY(i);
    const vz = posAttr.getZ(i);
    const d = (vx - pt.x) ** 2 + (vy - pt.y) ** 2 + (vz - pt.z) ** 2;
    if (d < minDist) {
        minDist = d;
        closestVert = [vx, vy, vz];
    }
}
geodesicSource = closestVert;  // 用最近顶点代替实际点击位置
```

**问题**：用户点击的精确位置信息丢失，测地线起止点只能是网格顶点，无法落在面片内部任意位置。

---

### 针对切片的补丁

**判断**：CGAL 切片对复杂模型会返回多个断开的轮廓段，需要把这些段拼接成完整的轮廓环。

**第一个补丁**：`join_contour_segments` 贪心拼接所有段

```python
def join_contour_segments(segments):
    # 贪心算法：把所有段按首尾距离拼接成一条线
    ...
```

**产生的新问题**：对复杂非凸模型（如怪物模型），手臂、腿等本来独立的轮廓段被强行拼接，出现横跨整个模型的长连线。

**第二个补丁**：加距离阈值，距离远的段不拼接

```python
if best_dist < tol:
    # 距离近才拼接
    ...
```

**问题**：每次打补丁都在给上一个补丁打补丁，根本原因始终没有解决。

## 三、第二阶段：发现根本原因

### 关键发现：凸包网格缺失面片

在可视化工具中检查生成的凸包 OBJ 文件时，发现网格明显缺少若干面片（标注区域有明显破洞）。

对 OBJ 文件做拓扑分析：

```python
from collections import defaultdict

edge_count = defaultdict(list)
for idx, (a, b, c) in enumerate(faces):
    for e in [(min(a,b),max(a,b)), (min(b,c),max(b,c)), (min(a,c),max(a,c))]:
        edge_count[e].append(idx)

# 找边界边（只被1个面片引用）
boundary = [e for e, faces in edge_count.items() if len(faces) == 1]
```

确认存在边界边，网格不封闭。

---

### 追溯到 C++ 底层：两次调用 `NewtonApple_hull_3D`

查看 `HGP_3D_Convex_Hulls_C2` 的 C++ 实现：

```cpp
// 第一次调用（结果正确）
int nx = de_duplicateR3(pts, outx, pts2);
int ts = NewtonApple_hull_3D(pts2, tris);

// 把第一次结果展开成点列表
for (int i = 0; i < (int)tris.size(); i++) {
    pts.push_back(pts2[tris[i].a]);
    pts.push_back(pts2[tris[i].b]);
    pts.push_back(pts2[tris[i].c]);
}

// 清空，对展开后的点再去重
pts2.clear(); outx.clear(); tris.clear();
nx = de_duplicateR3(pts, outx, pts2);

// 第二次调用（结果不完整）← bug 根源
ts = NewtonApple_hull_3D(pts2, tris);
```

**原始意图**：第一次调用可能产生法向不一致的面片，通过展开→去重→第二次调用来修正。

**实际后果**：第二次调用的输入是展开后再去重的点集，与原始点集存在精度差异（`R3` 结构体用 `float` 存储，而输入是 `double`，存在 `double→float→double` 的精度损失），导致 `NewtonApple_hull_3D` 在某些几何配置下输出不完整的凸包，漏掉若干面片。

---

### 完整的错误传播链

```
HGP_3D_Convex_Hulls_C2 缺失面片（NewtonApple 两次调用）
        ↓
save_mesh 写出的 OBJ/OFF 是不封闭网格
        ↓
CGAL 测地线算法在不封闭网格上崩溃或返回错误
        ↓
误判为"需要把点吸附到顶点"→ 前端加 snap 补丁
        ↓
CGAL 切片在破洞区域产生异常段
        ↓
误判为"需要拼接多段"→ 后端加 join_contour_segments
        ↓
复杂模型上出现长连线
        ↓
误判为"需要距离阈值"→ 再打一个补丁
```

## 四、第三阶段：根本修复

### 为什么不用 `repair_mesh`

`repair_mesh` 能合并重复顶点、删除退化面片、修复法向冲突，但**无法补全缺失的面片**。缺失面片是底层算法根本没有生成，修复函数拿不到不存在的数据。

### 为什么不用 `scipy.ConvexHull`

`scipy.spatial.ConvexHull` 基于 Qhull，可以在 Python 层正确生成凸包，但引入了外部依赖，违背本库"使用自有算法"的设计初衷。

### 最终方案：C++ 底层换用 CGAL `convex_hull_3`

`C1` 函数已经证明了 CGAL 的可靠性，直接将 `C2` 和 `C4` 的实现替换为 CGAL：

```cpp
extern "C" LIBHGP_EXPORT void HGP_3D_Convex_Hulls_C2(
    const Vector3d1 & vec,
    Vector3d1 & hull_points,
    Vector1i1 & hulls_surface_0,
    Vector1i1 & hulls_surface_1,
    Vector1i1 & hulls_surface_2)
{
    std::vector<Point_3> points;
    for (int i = 0; i < (int)vec.size(); i++)
        points.push_back(Point_3(vec[i][0], vec[i][1], vec[i][2]));

    if (points.size() <= 3) { hull_points = vec; return; }

    // CGAL 计算凸包，保证完整封闭 + 法向一致
    Polyhedron_3 poly;
    CGAL::convex_hull_3(points.begin(), points.end(), poly);

    // 建立顶点句柄→索引映射
    std::map<Polyhedron_3::Vertex_const_handle, int> vertex_index_map;
    int idx = 0;
    for (auto vit = poly.vertices_begin(); vit != poly.vertices_end(); ++vit, ++idx) {
        Point_3 p = vit->point();
        hull_points.push_back(Vector3d(
            CGAL::to_double(p.x()), CGAL::to_double(p.y()), CGAL::to_double(p.z())));
        vertex_index_map[vit] = idx;
    }

    // 导出面片索引
    for (auto fit = poly.facets_begin(); fit != poly.facets_end(); ++fit) {
        auto hc = fit->facet_begin();
        int a = vertex_index_map[hc->vertex()]; ++hc;
        int b = vertex_index_map[hc->vertex()]; ++hc;
        int c = vertex_index_map[hc->vertex()];
        hulls_surface_0.push_back(a);
        hulls_surface_1.push_back(b);
        hulls_surface_2.push_back(c);
    }
}
```

CGAL 的 `Polyhedron_3` 基于半边数据结构，天然保证：
- ✅ 面片完整封闭，不会缺失
- ✅ 法向统一朝外
- ✅ 严格流形拓扑，每条边被恰好两个面片共享
- ✅ 全程 `double` 精度，无损失

## 五、修复后的清理工作

### 后端清理

| 内容 | 处理 |
|---|---|
| `join_contour_segments` 函数 | 删除 |
| 凸包接口的 `repair=True` | 改为 `repair=False` |
| 测地线接口的 snap 补丁 | 删除 |

切片接口恢复为最简形式：

```python
@app.post("/api/slicer")
def slicer(req: SlicerRequest):
    mesh_path = get_mesh_path(req.mesh_id, ext="off")
    offsetses, offsets = hgp_py.HGP_Slicer_Mesh(
        mesh_path, req.plane_normal, req.plane_ds
    )
    return {
        "slices": [
            {
                "contours": [
                    [[p[0], p[1], p[2]] for p in seg]
                    for seg in plane_contours
                ]
            }
            for plane_contours in offsetses
        ]
    }
```

### 前端清理

**测地线选点**：删除 `closestVert` 遍历逻辑，直接使用 `hits[0].point`（射线与网格的精确交点）：

```javascript
const pt = hits[0].point;
const coord = [pt.x, pt.y, pt.z];  // 真实交点，不吸附到顶点
geodesicSource = coord;
```

**切片闭合**：保留距离阈值判断（对复杂非凸模型有意义），但明确其语义是"正常几何行为"而非补丁：

```javascript
// 凸包：首尾天然相连，自动闭合
// 复杂非凸模型：独立弧段，不强行闭合
if (pts[0].distanceTo(pts[pts.length - 1]) < 0.2) {
    pts.push(pts[0]);
}
```

## 七、经验教训

1. **表象问题不等于根本原因**：测地线崩溃和切片断裂看起来是两个独立问题，实际上同源于凸包缺失面片，沿着表象打补丁只会让代码越来越复杂。

2. **几何算法的错误具有传播性**：底层数据质量问题（不封闭网格）会向上传播，在所有依赖该数据的算法中都表现为"奇怪的错误"。

3. **先验证数据，再调试算法**：遇到几何算法异常，第一步应该验证输入数据的拓扑正确性（欧拉公式、边界边检测），而不是直接修改算法逻辑。

4. **补丁的代价**：每个补丁都在掩盖问题的同时引入新的约束，最终形成难以维护的代码。根本修复虽然成本高，但能同时解决所有下游问题，并且让代码回归简洁。
